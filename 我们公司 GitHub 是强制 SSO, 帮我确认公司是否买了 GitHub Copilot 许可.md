## 2) 立刻能做的“让 Cody 不那么傻”的最小动作清单（不求改制度）

### ✅ 1）先把“上下文”补齐（这是 80% 体验差的根因）

- 如果你现在在 VS Code 里问问题时，Cody 的回答经常“不贴你项目”，**优先怀疑仓库没被 indexing / 没抓到文件**。[[1\]](https://carcgl.sharepoint.com/sites/CodyTrialProgram561/_layouts/15/Doc.aspx?sourcedoc={1248793B-F909-468D-B628-79F7EC936FFE}&file=Can%2Bwe%2Bget%2BIndexed%2BRepositories%2Bin%2BSource%2BGraph%2Bso%2Bour%2Bplugins%2Bare%2Bsmarter_.doc&action=default&mobileredirect=true&DefaultItemOpen=1)[[2\]](https://carcgl.sharepoint.com/sites/CodyTrialProgram561/_layouts/15/Doc.aspx?sourcedoc={08F14993-EF59-4F31-BA41-DBF96B4618EB}&file=Can%2Bwe%2Bget%2BIndexed%2BRepositories%2Bin%2BSource%2BGraph%2Bso%2Bour%2Bplugins%2Bare%2Bsmarter_.docx&action=default&mobileredirect=true&DefaultItemOpen=1)
- 你们的 FAQ 也明确说：当 Cody 不准，常见原因是**它找不到相关文件作为 context**；并建议你查看“Context row”里实际用了哪些文件。[[7\]](https://carcgl.sharepoint.com/sites/CodyTrialProgram561/Shared Documents/General/Carrier Cody - Troubleshooting FAQs.pdf?web=1)
- 若仓库缺失，需要联系 **[swcoe@carrier.com](mailto:swcoe@carrier.com)** 走加仓/索引流程（FAQ 里写了联系渠道）。[[7\]](https://carcgl.sharepoint.com/sites/CodyTrialProgram561/Shared Documents/General/Carrier Cody - Troubleshooting FAQs.pdf?web=1)

> 你可以把它当作“Copilot 默认能看当前 repo”，而 Cody 在你们环境里**可能默认看不到**，需要先把 indexing 打开。

### ✅ 3）用 Cody 擅长的用法“卡住它的能力边界”

你们内部材料总结过：Cody **更擅长**

- 解释代码、生成单测、按“精确指令”做 edits、给多个实现思路；
   而对
- 现有架构大改、复杂推理、数学/数据推理
   “结果可能波动”。[[3\]](https://carcgl.sharepoint.com/sites/CodyTrialProgram561/Shared Documents/General/Carrier Cody - Capabilities and Training Videos.pdf?web=1)[[4\]](https://carcgl.sharepoint.com/sites/rlcchinasccoe/Shared Documents/Global Tools/Trainings/Cody/Carrier Cody - Capabilities and Training Videos.pdf?web=1)

所以策略是：**少让它“自由发挥”，多给它“可核对的约束”**。比如：

- “只改这个函数，保持接口不变”
- “按我给的伪码/步骤实现”
- “输出 diff + 每处变更理由”

这能显著减少“胡写”。
