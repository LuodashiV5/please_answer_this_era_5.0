#### 3. 语义密度 (Semantic Density)

技术协议里有很多缩写和专有名词。

## 1. 为什么转 Markdown 会有一堆 `NaN`？

`NaN`（Not a Number）在 Excel 转 Markdown 时通常出现在**合并单元格**、**空行**或**公式引用失败**的地方。

- **原因：** 大多数开源转换工具在处理 Excel 的复杂嵌套（比如跨行跨列）时，逻辑是“平铺直叙”的。一旦遇到合并单元格，它找不到对应的键值对，就只能吐出 `NaN` 或空白。
    
- **对策：** 不要指望 Dify 自带的上传插件能处理这种“硬骨头”。
    
    - **工程做法：** 先用 Python 的 `pandas` 库配合 `openpyxl` 进行预处理。
        
    - **核心逻辑：** `df.fillna(method='ffill')`（向前填充）。在嵌入式数据处理中这叫“信号保持”，在 Excel 转换里，这能把合并单元格产生的 `NaN` 自动补全为它原本该有的标题。

## 2. 如何验证 90+% 的转换有效率？

|---|---|---|
|**结构完整性**|正则表达式检查|检查 Markdown 表格的 `|
|**关键词覆盖**|逆向抽样|随机从原 Excel 抽 20 个冷门协议码，在转换后的 Markdown 里搜索，如果搜不到，直接判定这批数据转换失败。|
|**语义对齐**|LLM 抽检|给 AI 一个原文件片段和转换后的片段，问它：“语义是否有损？”这种抽检比人肉看要快得多。|

## 3. “混合检索 + Top K” 的调优方案

你觉得是在做模糊控制，是因为你现在手里的“控制参数”太多了（语义权重、关键词权重、重排分数阈值）。

要从“体力活”解脱，你需要从“手动修”转向“**小规模实验迭代**”：

- **第一阶段（比例设定）：** 初始设定关键词与向量比例为 **7:3**。因为协议文档（如：`0x55 AA`）这种强特征数据，关键词检索远比向量搜索精准。
    
- **第二阶段（引入 Rerank）：** 别只盯着 Top K。Top K 就像是 EXV 的粗调。你需要在后面加一个 **Rerank（重排模型）**，它就像是微调旋钮。
    
    - 先粗筛 50 个片段（Top 50），然后让 Rerank 模型（如 BGE-Reranker）把最相关的 5 个排到前面。**只要 Rerank 够强，你前面的 Top K 设大一点也无所谓。**
