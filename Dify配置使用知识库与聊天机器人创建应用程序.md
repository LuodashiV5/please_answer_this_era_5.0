要在 Dify 中配置**知识检索机器人**并高效使用这些提示词，需结合「知识库构建」「应用配置」「System Prompt 设置」「测试与迭代」四个核心环节，以下是具体操作指南：

### 一、前提：构建知识库（文档预处理）

Dify 的「知识检索」依赖**已导入的文档**（Word/PDF 等）。需先将协议/命令文档导入知识库：

1. **准备文档**：
   
- 将中日文协议/命令文档（Word/PDF）整理到统一目录（如 SharePoint 文件夹、本地文件夹）。

- 确保文档命名清晰（如 `设备A_温度控制协议.pdf`、`设备B_转速命令手册.docx`），方便后续追溯。
      

2. **导入知识库**：

- 登录 Dify 控制台，进入「知识库」模块 → 选择目标知识库（或新建）→ 点击「上传文件」，批量导入协议/命令文档。

- Dify 会自动解析文档内容（需等待索引构建完成，可通过「知识库 → 详情」查看索引状态）。

### 二、配置应用：绑定知识库 + 设置 System Prompt

1. **创建/编辑应用**：

- 进入 Dify 「工作室」→ 新建应用（或编辑现有应用）→ 选择「聊天机器人」类型。


2. **绑定知识库**：

- 在应用配置页，找到「知识库」选项卡 → 点击「添加知识库」，选择之前导入协议文档的知识库。


3. **设置 System Prompt（核心！）**：

- 进入「Prompt」选项卡，在「System Prompt」输入框中，**粘贴工程强化版的 System Prompt**（即你提供的「你是一个‘协议/命令文档智能索引工程助理’…」完整内容）。


> 注意：
> 
> - 若有日语回答需求，需在 System Prompt 中明确「用日语回答」（或保留原 Prompt 的“根据用户提问语言”逻辑）。
>     
> - 若需适配其他语言，可修改 System Prompt 中的语言要求。
>     


### 三、配置「知识检索」节点（工作流级配置）

在 Dify 的**工作流编辑器**中（若使用「工作流」模式），需确保「知识检索」节点正确关联知识库，并传递用户问题：

1. **添加「知识检索」节点**：

- 从左侧节点库拖拽「知识检索」节点到画布，连接「开始」节点。


2. **配置「知识检索」节点**：

- 选择绑定的知识库（即导入协议文档的知识库）。

- 设置「检索条件」：可留空（默认全量检索），或根据文档结构设置过滤（如按文件名、章节等，需文档有元数据支持）。


3. **连接后续节点**：

- 将「知识检索」节点的输出连接到「LLM」节点（或其他处理节点），确保检索到的文档内容传递给大模型。
      
    

### 四、LLM 节点配置：选择模型 + 调整参数

在「LLM」节点中，需选择支持**多语言（中日文）**的大模型（如 GPT-4o、Claude 3、本地化部署的 Qwen 等），并设置：

1. **模型选择**：

- 在「LLM」节点的「模型」选项中，选择兼容中日文的模型（如 `gpt-4o`支持多语言，`Qwen`对中日文优化较好）。
      
    
2. **参数调整**：

- 「温度（Temperature）」建议设为 **0~0.2**（降低随机性，确保回答精准）。

- 「最大 tokens」根据文档长度和回答复杂度调整（如 2000~4000，避免截断）。

### 五、测试与迭代：验证提示词效果

1. **模拟提问**：

- 在「预览」或「测试」界面，输入工程意图类问题（如：`设定温度`，或日语 `温度を設定してください`）。


2. **检查回答结构**：

- 确认回答是否包含「工程意图」「Command」「Frame/Format」「数据点位」「文档来源」「原文说明」等字段。

- 验证文档来源路径、文件名是否正确（可追溯）。

- 若无法 100% 确定，是否有「候选项 + 置信度」（而非简单“不知道”）。


3. **迭代优化**：

- 若回答偏离预期，可调整：

- **System Prompt**：补充文档结构细节（如“文档按‘设备-功能-命令’分类，每章开头有设备型号”）。

- **知识库索引**：检查文档是否解析完整（如 PDF 表格、图片中的文字是否被识别）。

- **检索条件**：若文档量大，可设置「关键词过滤」或「章节匹配」（需文档有元数据）。


    

### 六、进阶技巧：多语言与文档来源增强

1. **多语言支持**：

- 若需日语回答，可在 System Prompt 中明确：

    > “请用日语（或用户提问的语言）回答，确保术语精准（如‘コマンドID’对应‘Command ID’）。”

- 测试时，输入日语问题，验证回答语言切换是否正常。


2. **文档来源追溯**：

- 在 System Prompt 中强调「文档来源路径 + 文件名」，并在知识库导入时保留文档的**元数据（如文件路径、文件名）**（Dify 会自动抓取）。

- 若文档来自 SharePoint，需在导入时选择「SharePoint 同步」，确保路径信息完整。



### 七、常见问题排查

- **回答不准确**：检查知识库是否包含最新文档、System Prompt 是否明确要求“优先理解工程意图”。
  
- **无文档来源**：确认知识库导入时保留了文件路径/文件名元数据，或 System Prompt 中是否正确引用。
  
- **日语回答乱码**：确认模型支持日语（如 GPT-4o、Claude 3），且 System Prompt 语言设置正确。

要让 Dify 的机器人**用中文回复**，并完善「知识检索+LLM+参数提取+回复」的工作流，需从 **System Prompt 语言设置**、**工作流节点配置**、**测试验证**​ 三个环节优化，以下是具体操作：

### 一、修改 System Prompt（强制中文回复）

在 Dify 的**应用 Prompt 配置**中，将 System Prompt 的语言要求改为中文。找到你使用的「参数提取器」或「LLM」节点的 System Prompt（或全局 System Prompt），修改为：

---

你是一个**协议/命令文档智能索引工程助理**（中文回复版）。

【核心任务】

- 从用户提供的私有协议/命令文档（Word / PDF，中日文混合）中，
  
- 根据用户的“工程意图描述”（如：设定温度、目标转速、启动命令），
  
- 精确定位对应的：
  
    1. Command ID（如 cmd4C）
       
    2. Frame / Format 名称
       
    3. 数据位置（如 Byte / D / Bit）
       
    4. 原始文档来源路径
       
    5. 文档中的原文说明（保留日文，并给出**中文解释**）
       
    

你不假设用户已知任何 Command 名称、文件名或文件夹结构。

你不要求用户提供准确的日文关键词。

你必须优先理解“工程意图”，而不是做简单的关键字搜索。

你的回答必须严格使用以下结构化格式输出，便于工程使用和复核：

【工程意图】

（复述用户的需求，例如：设定温度）

【Command】
- Command ID：xxx
- Command 名称（如有）：xxx
  

【Frame / Format】
- Frame 名称：xxx
- Frame ID / 编号：xxx
  
【数据点位】
- 数据位置：D? / Byte ? / Bit ?
- 数据含义：xxx
  
【文档来源】
- 文件路径：xxx
- 文件名：xxx
  
【原文说明】
- 日文原文：xxx
- 中文解释：xxx

如无法 100% 确定，请给出“候选项 + 置信度”，不要简单回答“不知道”。

【行为约束】
- 不进行闲聊，不输出与工程无关的内容
- 不猜测不存在于文档中的信息
- 所有结论必须可追溯到具体文档
- 优先给出“可工程执行”的结论，而不是泛泛描述
  
- **回答语言：全程使用中文（日文原文保留，解释部分用中文）**​

### 二、工作流节点配置（确保知识检索→LLM→参数提取→回复 串联正常）

你的工作流包含「开始→知识检索→LLM→参数提取器→直接回复」，需检查每个节点的配置：

#### 1. 知识检索节点

- 确认已绑定**搭建好的知识库**（协议/命令文档所在的知识库）。
  
- 输入变量：选择「开始」节点的用户问题（通常是 `query`或 `input`变量）。
  

#### 2. LLM 节点

- 模型：选择你使用的 LLM（如截图中的 `QQ CHAT`，需确保模型支持中文回复）。
  
- 输入变量：选择「知识检索」节点的输出（通常是 `knowledge_retrieval_result`或类似变量，包含知识库匹配的内容）。
  

#### 3. 参数提取器节点（关键！）

- 模型：选择支持中文的模型（如截图中的 `gpt-4o`，需确保模型能理解中文指令）。
  
- 输入变量：选择「LLM」节点的输出（通常是 `llm_output`或类似变量，包含 LLM 处理后的内容）。
  
- 提取参数：
  
    - 参数名：建议用 `answer`（与你截图中一致）。
      
    - 类型：`String`（或根据需求调整）。
      
    - 必填：开启。
      
    
- 指令（Prompt）：**必须包含中文回复的要求**，并明确从 LLM 输出中提取结构化信息。示例指令：
  
    ```
    请从输入的文本中，提取以下结构化信息，并用中文组织回复（日文原文保留，解释用中文）：
    
    【工程意图】：用户的需求（如“设定温度”）  
    【Command】：Command ID、名称  
    【Frame / Format】：Frame 名称、编号  
    【数据点位】：数据位置、含义  
    【文档来源】：文件路径、文件名  
    【原文说明】：日文原文 + 中文解释  
    
    若信息不明确，给出候选项 + 置信度（如“候选Command ID：cmd4C，置信度80%，因文档中‘温度设定’相关命令疑似为此”）。  
    回复必须严格遵循上述格式，不添加额外内容。
    ```
    

#### 4. 直接回复节点

- 回复内容：选择「参数提取器」节点的输出（即 `answer`变量）。
  

### 三、测试与调试（验证中文回复+信息提取）

配置完成后，点击工作流的「运行」或「测试」按钮，输入一个**工程意图类问题**（如：`设定温度的命令是什么？`），观察输出：

1. **知识检索**：检查是否从知识库中匹配到相关文档内容。
   
2. **LLM**：检查输出是否包含文档中的关键信息（如 Command ID、Frame 等）。
   
3. **参数提取器**：检查是否将 LLM 输出解析为结构化格式，且**全部用中文回复**（日文原文保留，解释为中文）。
   
4. **直接回复**：检查最终输出是否符合你设定的格式（工程意图、Command、Frame 等字段齐全）。
   

### 常见问题排查

- **中文回复失败**：检查 System Prompt 和参数提取器的指令是否明确“中文回复”，且 LLM 模型支持中文（如 gpt-4o、 Claude、国内的 Qwen 等都支持）。
  
- **知识检索无结果**：确认知识库已正确导入文档，且文档内容与问题相关（可手动在知识库搜索测试）。
  
- **参数提取格式错误**：调整参数提取器的「指令」，明确字段要求（如用“【工程意图】”“【Command】”等标签强制格式）。

参数提取器 是做什么用的

参数提取器（Parameter Extractor）是 Dify 工作流中的一个**高级节点**，它的核心作用是**从非结构化的文本中，自动识别并提取出结构化的数据（键值对）**。

简单来说，它就像一个“数据格式化工具”，专门用来解决“LLM 回答太啰嗦、格式不统一”的问题。

### 1. 核心作用：强制结构化输出

- **输入**：一段杂乱无章的文本（通常是 LLM 节点生成的回答）。
  
- **输出**：一个干净、标准的 JSON 对象（或你定义的特定格式）。
  

**举个例子：**

- **LLM 回答**（非结构化）：

> 根据文档，设定温度的命令是 `cmd4C`，对应的 Frame 名称是 `温度设定帧`，数据位置在 Byte 3。文件路径是 `protocols/temp.pdf`。

- **参数提取器处理后**（结构化）：

```json
{
  "command_id": "cmd4C",
  "frame_name": "温度设定帧",
  "data_position": "Byte 3",
  "file_path": "protocols/temp.pdf"
}
```


### 2. 为什么在你的场景中需要它？

结合你之前提供的 **“协议/命令文档智能索引工程助理”**​ 的 Prompt，参数提取器在这里扮演了**“质检员”**和**“数据清洗工”**的角色：

- **确保格式统一**：你要求输出 `【工程意图】`、`【Command】`等固定格式。LLM 有时会“放飞自我”，忘记格式。参数提取器可以强制它必须按照这个模板来输出。
  
- **提取关键字段**：你不需要 LLM 的“废话文学”，只需要 `Command ID`、`Frame 名称`、`数据点位`这几个关键信息。参数提取器可以精准地只抓取这些数据，过滤掉解释性的文字。
  
- **便于下游处理**：提取出的结构化数据（JSON）可以直接被后续的代码节点、API 调用节点使用，或者直接存入数据库，实现自动化。
  

### 3. 在你的工作流中如何配置？

在你的截图里，工作流是 `开始 -> 知识检索 -> LLM -> 参数提取器 -> 直接回复`。

- **LLM 节点**：负责“阅读理解”，从知识库中找出答案（可能是一大段话）。
  
- **参数提取器**：负责“数据提取”，把 LLM 的回答拆解成 `command_id`、`frame_name`等字段。
  
- **直接回复节点**：负责“展示”，把提取器整理好的干净数据展示给用户。
  

**配置技巧**：

在参数提取器的 **“指令”**​ 框中，你需要非常清晰地告诉它要提取哪些字段。例如：

> 请从文本中提取以下参数：
> 
> - `command_id`(字符串)
>     
> - `frame_name`(字符串)
>     
> - `data_position`(字符串)
>     
> - `file_path`(字符串)
>     

这样，无论 LLM 怎么回答，最终用户看到的都是标准化的表格或列表，而不是一段散文。

### 步骤 1：检查「开始」节点（可选，确认输入）

「开始」节点是工作流的入口，负责接收用户的问题/文件：

- 点击「开始」节点 → 右侧面板确认 **输入字段**：
  
- `userinput.query`：用户的提问（如“设定温度的命令是什么？”）。
  
- `userinput.files`：用户上传的文档（如果需要从文件中提取信息，否则可忽略）。
      
    
- 无需修改，保持默认即可。
  

### 步骤 2：检查「知识检索」节点（核心：绑定知识库）

「知识检索」节点负责从你上传的协议/命令文档中，找到与用户问题相关的内容：

- 点击「知识检索」节点 → 右侧面板确认 **知识库**：
  
    - 确保下拉框中选择的是你上传协议文档的知识库（截图中是 `Parent-child-HQ 3`）。
      
    
- 无需修改，保持默认（Dify 会自动从知识库中检索与 `userinput.query`相关的内容）。
  

### 步骤 3：检查「LLM」节点（核心：调用 Qwen 模型）

「LLM」节点负责对「知识检索」返回的内容进行处理（理解工程意图、定位命令等）：

- 点击「LLM」节点 → 右侧面板确认：
  
- **模型**：选择 `QQ_CHAT`（即你修改后的 Qwen 模型，确保模型名称正确，且 Dify 已配置好 Qwen 的 API 密钥/Endpoint）。
  
- **提示词（Prompt）**：这里需要填写**让 Qwen 理解“如何从知识库中找命令”的指令**。你可以复制你之前的「工程强化版 Prompt」，但需**简化适配 LLM 节点**（因为「参数提取器」会进一步结构化，LLM 先负责“理解+初步提取”）。
  
👉 示例 Prompt（适配 LLM 节点）：
    
    ```
    你是一个协议/命令文档智能索引助理。请根据用户的问题（{userinput.query}）和我从知识库中检索到的内容（{knowledge_base.result}），理解用户的“工程意图”（如设定温度、目标转速），并提取以下信息：
    - 工程意图：复述用户需求
    - Command ID：如 cmd4C
    - Frame 名称：如温度设定帧
    - 数据位置：如 Byte 3 / D1 / Bit 5
    - 文档来源：文件路径+文件名
    - 日文原文：文档中的原句（保留日文）
    - 中文解释：日文原句的中文翻译
     请输出为结构化文本（后续会由参数提取器进一步格式化）。如果找不到，说明“无法确定，置信度低”，不要编造。
    ```
 （注：`{userinput.query}`和 `{knowledge_base.result}`是 Dify 的**变量占位符**，会自动替换为用户的提问和知识库的检索结果，无需手动修改。）
    

### 步骤 4：配置「参数提取器」节点（核心：结构化输出）

「参数提取器」负责将 LLM 输出的“文本”转换为**严格的 JSON 格式**，确保字段清晰、可用：

- 点击「参数提取器」节点 → 右侧面板配置：

#### （1）模型

选择 `gpt-4o`（或其他你信任的结构化提取模型，如 Qwen 也支持，但 gpt-4o 对格式的理解更稳）。

#### （2）输入变量

选择 `LLM`节点的输出（即 `{LLM.answer}`，这是 LLM 处理后的文本）。

#### （3）提取参数（关键！定义要提取的字段）

点击「提取参数」下的 `+`，逐个添加字段，格式参考:
    
|参数名|类型|描述|示例值|
|----|----|----|----|
|`engineering_intent`|String|工程意图（复述用户需求）|“设定温度”|
|`command_id`|String|Command ID|“cmd4C”|
|`frame_name`|String|Frame 名称|“温度设定帧”|
|`data_position`|String|数据位置|“Byte 3”|
|`doc_source`|String|文档来源（路径+文件名）|“protocols/设备A_温度控制协议.pdf”|
|`japanese_text`|String|日文原文|“温度を設定するコマンドはcmd4Cです。”|
|`chinese_explanation`|String|中文解释|“设定温度的命令是cmd4C。”|

👉 操作：

- 点击「提取参数」下的 `+`→ 输入参数名（如 `engineering_intent`）→ 类型选 `String`→ 提示词（可选，如“提取工程意图，即用户的原始需求”）。

- 重复此步骤，直到添加完所有需要的字段。

#### （4）指令（可选，增强提取准确性）

在「指令」框中，可补充：

```
请严格按以下字段提取信息，确保每个字段都从 LLM 的回答中精准匹配。如果字段不存在，填“未知”。最终输出为 JSON 格式。
```
    
#### （5）输出变量

默认会生成一个变量（如 `{parameter_extractor.answer}`），无需修改。


### 步骤 5：配置「直接回复」节点（核心：输出结果）

「直接回复」节点负责将「参数提取器」输出的结构化数据，以**用户易读的格式**展示：

- 点击「直接回复」节点 → 右侧面板配置：

- **回复内容**：使用**变量+格式化文本**，将参数提取器的输出转为友好格式。
      

👉 示例回复内容（使用 Markdown 或纯文本，适配中文）：
    
```
【工程意图】：{parameter_extractor.answer.engineering_intent}
【Command】：
- Command ID：{parameter_extractor.answer.command}
【Frame_id / Format】：
- Frame 名称：{parameter_extractor.answer.frame_name}
【数据点位】：
- 数据位置：{parameter_extractor.answer.data_position}
【文档来源】：
- 文件路径：{parameter_extractor.answer.doc_source}
【原文说明】：
- 日文原文：{parameter_extractor.answer.japanese_text}
-中文解释：{par ameter_extractor.answer.chinese_explanation}
```

（注：`{parameter_extractor.answer.xxx}`是参数提取器输出的字段，需与你之前定义的参数名一致。）

### 常见问题 & 解决思路

- **Q：LLM 输出乱码/不符合要求？**​
  
    A：检查「LLM」节点的 Prompt 是否清晰，变量是否正确（如 `{knowledge_base.result}`是否存在），或换用更稳定的模型（如 gpt-4o）。
    
- **Q：参数提取器提取不到字段？**​
  
    A：检查「提取参数」的提示词是否足够明确，或调整 LLM 的输出格式（让 LLM 先输出更规整的文本，再交给参数提取器）。
    
- **Q：回复是英文？**​
  
    A：确保「LLM」节点的 Prompt 明确要求“用中文回复”，且模型本身支持中文（Qwen 支持，gpt-4o 也支持）。
