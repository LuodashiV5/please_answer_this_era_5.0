## 一、我昨天在“学什么”

  - Agent / Skill / Tag 的关系
  - Agent 的创建入口
  - Agent 的可见性（个人 / App / 组织）

👉 **这是一次从“文本 Prompt 设计”走向“平台能力落地”的关键跃迁尝试**

### 2️⃣ 实际动手创建 Copilot Agent

- 我尝试创建了 **「技术负责人 Copilot Agent」**
- 给 Agent 定义了：
  - 角色（Tech Lead / Embedded Architect）

**但随后遇到强烈落差：**

- 创建完成后：

  - ❓ 在哪里能再次找到它？

  - ❓ 后续对话是不是应该都和它进行？

## 二、关键困惑与真实认知冲突

### ⚠️ 困惑 1：**Skill ≠ Agent（不是一一映射）**

- 我一开始默认：

  > 一个 Skill.md ≈ 一个 Copilot Agent

- 但现实是：

  - **Skill 是“能力描述 / Prompt 资产”**
  - **Agent 是“运行时载体 / 产品对象”**

- 多个 Skill 更合理的状态是：

  - 👉 **作为 Agent 的内部能力库**
  - 👉 或作为“使用时手动引用的 Prompt 模板”

✅ 这是一次**模型从“文件思维”到“产品思维”**的修正

------

### ⚠️ 困惑 2：个人创建 ≠ 组织可见

- 我发现一个重要事实（隐含但非常关键）：
  - **个人创建的 Agent，默认只存在于创建上下文**
  - 并不会：
    - 自动出现在 Copilot App 列表
    - 自动成为“长期可调用实体”
- 这解释了：
  - “找不到 Agent”
  - “感觉像没保存”

👉 **这不是我操作错，而是平台治理与权限模型的设计结果**

------

### ⚠️ 困惑 3：Agent ≠ 日常对话的唯一入口

- 我一度期望：

  > “以后是不是就只跟这个 Agent 聊了？”

- 实际更合理的模型是：

  - **Copilot 主对话 = 通用入口**
  - **Agent = 特定场景的强化模式**
  - **Skill = 可随时注入的能力模块**

✅ 这让我重新理解了：
 **Agent 更像“模式”，不是“新 Copilot 本体”**

------

## 三、昨天最大的学习收获（认知层）

### ✅ 收获 1：我做的是“能力工程化”，不是玩 Prompt

- 我过去写的 SKILL.md：不是零散 Prompt; 而是：有边界; 有 DoD; 有输入输出约束
- 昨天我意识到：这本质上是在做 **“个人工程能力的 AI 封装”**

这是非常稀缺、且**长期可复用**的资产。

------

### ✅ 收获 2：平台限制 ≠ 价值不存在

- Copilot 当前：对个人 Agent 的发布、持久化、可见性限制较多
- 但这不否定：Skill 设计的价值; Prompt 资产的复利效应

✅ **能力先成熟，平台迟早会补齐**

------

### ✅ 收获 3：我需要的是“Agent 使用策略”，不是更多 Skill

- 我已经不是缺 Skill
- 而是需要：
  - 哪类 Agent 值得“长期存在”
  - 哪些 Skill 只作为“临时调用模板”
  - 哪些应该合并成一个“角色级 Agent”

------

## 四、对我自己的下一步启发（非行动项）

> 这是成长笔记，不是待办，但我记录下了“方向感”

- 把 Skill 分三类：
  1. **角色级能力包**（Tech Lead / Architect）
  2. **流程型 Skill**（日报、会议、任务拆解）
  3. **工具型 Skill**（代码审计、日志分析）
- 不再追求：
  - “每个 Skill 都变成 Agent”
- 而是追求：
  - **一个 Agent，能否自然调用我沉淀的能力体系**
