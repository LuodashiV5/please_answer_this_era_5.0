# SKILL: Code Review（致命风险优先）

> 给 Claude Code 使用：对我贴出的 C/C++/嵌入式代码做 **致命风险优先** 的 code review。

## 目标
只找 **会导致现场事故/宕机/越界/溢出/空指针/未初始化/竞态/死锁/硬件误动作** 这类“致命问题”，并给出可落地的改法。

## 输入
- 我将提供：代码片段或 diff（可能含宏、寄存器、ISR、RTOS、通信栈等）。

## 输出（必须严格遵守）
1) **只输出“改善建议”**，不要解释过程、不要复述需求、不要礼貌性客套。
2) **在代码旁逐行中文旁注**：
   - 用 `// REVIEW[BLOCKER]: ...` 标注致命问题
   - 只在确实有风险的行旁注；无问题的行不要注释刷屏
3) **必须给出最小修改方案**：
   - 优先提供 *补丁式* 修改（伪 diff 即可）
   - 不要大重构，除非无法规避致命风险
4) **每条建议必须包含**：
   - 风险点（1 句话）
   - 触发条件（1 句话，尽量可复现）
   - 修复方式（1~3 个可执行动作）
5) **重点检查**（按优先级）：
   - 越界访问（数组/指针/下标/长度）
   - 整型溢出/下溢（含位移、乘加、类型提升）
   - 空指针/野指针/生命周期
   - 并发（ISR/任务/多核/共享变量/volatile/原子性）
   - 宏多次求值/运算符优先级/副作用
   - 未定义行为（UB）：移位、对齐、别名、未初始化、越界、除 0

## 严格禁止
- 不要给“表扬”“总结”“背景科普”。
- 不要输出“建议你……先做……然后……”。直接给修改建议和旁注。

## 输出示例（格式示意）
```c
x = a[i]; // REVIEW[BLOCKER]: i 未校验，可能越界读，现场随机故障。修复：在进入前校验 i < a_len；或改用安全访问函数。
```
