# SKILL: Code Review（CERT C 致命风险版）

> **给 Claude Code 使用**：对我贴出的 C / 嵌入式 C 代码进行 **CERT C 视角的致命风险 Code Review**。
>
> 目标不是“风格/整洁”，而是 **防止崩溃、未定义行为、可被利用的漏洞**。

---

## 🎯 核心目标

**只聚焦 CERT C 中的“高危 / 可导致现场事故或漏洞”的问题**，包括但不限于：

- 空指针 / 野指针 / 生命周期错误
- 数组越界 / 指针越界
- 整型溢出 / 下溢 / 移位 UB
- 未初始化读
- 宏副作用 / 多次求值
- 并发数据竞争 / 非原子读改写
- 明确的未定义行为（UB）

> ✅ 优先参考 **CERT C Rules**，必要时补充 **CERT Recommendations**。

---

## 📥 输入

- 一段或多段 **C / 嵌入式 C 代码**（可包含宏、ISR、RTOS、多核共享变量）
- 可能是旧代码 / 现场问题代码 / 非 MISRA 风格代码

---

## 📤 输出（必须严格遵守）

### 1️⃣ 只输出「改善建议」，**不输出分析过程**

- ❌ 不要解释 CERT 是什么
- ❌ 不要教育用户
- ✅ 直接指出 **哪里危险 + 怎么改**

---

### 2️⃣ **逐行中文旁注（核心要求）**

在原代码旁，用如下格式：

```c
ptr->x = y;  // REVIEW[BLOCKER][EXP34-C]: ptr 可能为 NULL，直接解引用会崩溃。修复：先判空或接口保证 non-null。
```

#### 标注规范

- `REVIEW[BLOCKER]`：**必改，可能导致 crash / UB / 漏洞**
- `REVIEW[MAJOR]`：高风险，强烈建议修改

并 **必须带 CERT 规则号**：

- `EXPxx-C` 表达式 / 空指针 / UB
- `ARRxx-C` 数组 / 指针越界
- `INTxx-C` 整型 / 移位 / 溢出
- `MEMxx-C` 内存管理
- `CONxx-C` 并发 / 数据竞争
- `PRExx-C` 宏 / 预处理

---

### 3️⃣ 每一条问题，必须包含 3 个要素

- **风险点**：一句话说明会出什么事
- **触发条件**：在什么情况下会炸
- **修复方式**：给出可直接落地的修改方案

> ❌ 不要只说“建议检查”“注意风险”

---

### 4️⃣ 必须给出「最小修改方案」

- ✅ 优先给 **局部 if / 边界检查 / 类型修正**
- ❌ 不要直接建议“大重构”，除非这是唯一解
- ✅ 允许给 `assert / static_assert` 固化假设

---

## 🔍 CERT C 致命风险重点检查清单（按优先级）

### 🟥 第一优先：必炸 / 必 UB

- `EXP34-C`：空指针解引用
- `ARR30-C`：数组 / 指针越界
- `INT32-C`：有符号整型溢出
- `INT34-C`：非法移位（负数 / >= 位宽）
- `EXP33-C`：未初始化变量读取

---

### 🟧 第二优先：高概率踩雷

- `PRE31-C`：宏参数有副作用 / 多次求值
- `INT31-C`：隐式类型转换导致截断
- `CON30~43-C`：多线程 / ISR 数据竞争
- `EXP45-C`：条件表达式中赋值

---

## ✅ 推荐输出结构（强制）

```
A. 原代码 + 逐行旁注（CERT C）
B. 最小修复补丁（diff 或示例代码）
C. （可选）进一步加固建议
```

---

## ✅ 示例（风格参考）

```c
value = array[idx];
// REVIEW[BLOCKER][ARR30-C]: idx 未校验，可能越界读导致随机崩溃。
// 触发条件：idx >= ARRAY_LEN 或 idx 为负。
// 修复：在访问前校验 idx < ARRAY_LEN；或封装为安全访问函数。
```

```c
if (a + b > c)
// REVIEW[BLOCKER][INT32-C]: a+b 可能发生有符号溢出，行为未定义。
// 修复：改为 if (a > c - b) 或使用更宽类型计算。
```

- 不要输出总结性废话
- 不要输出“整体还不错”
- 不要把 MISRA 风格问题当成重点
- 不要遗漏 CERT 规则号
