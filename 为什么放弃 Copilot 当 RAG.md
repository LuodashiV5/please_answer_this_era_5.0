
# 为什么放弃 Copilot 当 RAG
  
## 结论先行
M365 Copilot / Copilot Studio **不适合作为工程级 RAG 引擎**，
不适合用于「协议 / 命令 / 点位」这类**强精确性**问题。
  
---
  
## 原因一：Copilot 的知识模型不是向量 RAG
- Copilot 知识库本质是 **索引 + 摘要**
- 不是 chunk 级向量召回
- 无法保证“命中哪一段原文”
  

  
---
  
## 原因二：无法稳定回答“工程型问题”
例如：
- cmd4C 的 D2 在哪里？
- 某个 bit 的定义来源于哪份文档？
  
表现问题：
- 回答不稳定
- 容易泛化
- 无法给出明确证据链
  
---
  
## 原因三：能力边界不可控
- 无法控制 embedding / chunk 策略
- 无法调试召回结果
- Graph API 不是为 RAG 设计
  
---
  
## 对比：为什么转向自建 RAG
- 可控的 embedding（Jina）
- 可控的向量库（Weaviate）
- 可调试的召回链路（Dify）
  

  
---
  
## 最终收敛
基于准确性、可控性、工程可解释性考虑，
放弃 Copilot 作为 RAG 引擎，
仅保留其作为 **UI / 助手入口** 的可能性。
