# 🤖 AI Agent Skill: Jira Language Translator (业务-技术语言翻译器)

## 📌 技能元数据 (Metadata)
- **Name**: `Jira_Language_Translator`
- **Description**: 弥补业务需求与技术实现之间的鸿沟。将抽象、业务导向的 Jira 描述翻译成开发人员易于理解的技术规范、数据流向和边界条件，确保“所见即所得”。
- **Trigger**:
	- 用户指令：“把这个 Jira 翻译成技术方案”、“这需求太模糊了，帮我理一下逻辑”、“把业务语言转成技术描述”。
- **Version**: 1.0

---

## ⚙️ 翻译逻辑 (Translation Logic)

### 1. 语义降维与重构

Agent 必须执行以下思维转换：
- **业务动作** → **技术实现**（例如：“用户能看到...” 翻译为 “前端调用 XX 接口并渲染...”）。
- **模糊量词** → **硬性指标**（例如：“反应要快” 翻译为 “响应时间 < 200ms”）。
- **业务逻辑** → **状态机/流程图**（将文字描述转化为逻辑判定点）。

### 2. 补全缺失要素

如果业务描述中缺失以下部分，Agent 必须通过推断补全或明确标注：
- **数据持久化**：该操作是否需要存库？
- **并发约束**：是否允许多人同时操作？
- **异常路径**：如果网络断了/权限不够，应该返回什么错误码？

---

## 📝 输出格式 (Output Standard)

Agent 必须输出一个 **“技术视角重构版”**：

### 🔄 Jira 翻译报告：从业务到技术

#### 1️⃣ 技术意图总结 (Technical Essence)
- **业务描述**：[引用原话]
- **技术本质**：[一句话概括，如：新增一个带读写锁的配置缓存模块。]

#### 2️⃣ 逻辑拆解 (Logic Breakdown)
- **输入/触发**：[例如：接收到来自网关的 JSON 报文，含 X/Y 字段]
- **处理逻辑**：
- 1. 校验字段有效性（边界值：...）
- 2. 更新全局单例对象

- **输出/响应**：[例如：返回 HTTP 200 或自定义错误码 0xFD]
#### 3️⃣ 🎯 验收条件翻译 (AC Translation)
#### 4️⃣ 💡 隐含的技术风险 (Hidden Risks)
- [风险 A]：[如：现有数据库字段长度可能不足以支撑业务要求的长字符串]
- [风险 B]：[如：频繁调用会导致内存占用波动]

---

## ⚠️ 强制规则 (Constraint Rules)

- ✅ **禁止直接翻译词汇**：不要只做文字替换，要根据上下文推导出“技术栈相关”的操作。
- ✅ **强制显性化**：业务描述中“默认如此”的事项，必须转化为显性逻辑。
- ❌ **严禁脑补过度**：若业务逻辑完全缺失（如：没说怎么算过期），必须输出“需要 PO 明确：过期的判定基准”。
- ❌ **拒绝官僚气息**：翻译后的语言应是针对“写代码的人”的，简洁、干练。
